🎯 Q1 · Architecture Fundamentals
"Walk me through how you'd design a system architecture for a new web application that needs to support 10,000 concurrent users."
💡 Tip: Structure your answer around: Presentation → Application → Data layers. Mention scalability from the start.
Start with a load balancer distributing traffic across multiple web servers (horizontal scaling). Use auto-scaling groups so the system expands under load and contracts to save cost. Separate the application layer from the database. Use a managed relational DB (RDS/Cloud SQL) for transactional data, with read replicas to distribute read load. Add a caching layer (Redis/Memcached) to reduce DB hits for hot data. Use a CDN to serve static assets. Finally — add monitoring (CloudWatch/Datadog) and ensure you have circuit breakers to prevent cascade failures. The key is to make each layer independently scalable.
🎯 Q2 · Trade-offs
"Explain the CAP theorem and describe a scenario where you had to choose between consistency and availability."
💡 Tip: Don't just define it — give a concrete example. E-commerce checkout vs product catalog behave differently.
CAP Theorem states that in a distributed system, you can only guarantee two of three: Consistency (all nodes see the same data), Availability (system always responds), or Partition Tolerance (system works despite network splits). Since network partitions happen in real distributed systems, you're really choosing between CP and AP. Example: For a payment service, I'd choose CP — it's better to refuse a transaction than to process a double charge. For a product catalog or social feed, I'd choose AP — stale data for a few seconds is acceptable; an outage is not. The key interview answer is showing you understand the business context drives the choice.
🎯 Q3 · Well-Architected
"How would you evaluate whether an existing architecture is 'good'? What framework or criteria do you use?"
💡 Tip: Reference the Well-Architected Framework pillars. Show structured thinking, not gut feel.
I use the AWS Well-Architected Framework as a structured lens: Operational Excellence (can we deploy and monitor reliably?), Security (are we protecting data and following least-privilege?), Reliability (does it recover from failure automatically?), Performance Efficiency (are we using resources appropriately for the load?), Cost Optimization (are we paying for what we actually need?), and Sustainability (are we minimizing environmental impact?). I'd run a Well-Architected Review with the team — interviewing stakeholders, reviewing architecture diagrams, checking IaC configs — and produce a findings report with prioritized remediation items. Interviewers love this answer because it shows process, not just instinct.

📦 Phase 2 · Cloud Foundations
IaaS, PaaS, SaaS & FaaS
The four cloud service models — and when each one is the right call.

Why This Decision Matters More Than People Realise

Choosing the wrong service model is one of the most expensive architectural mistakes you can make — not because the technology itself is wrong, but because the operational consequences compound over time. Pick IaaS when you should have picked PaaS and your team spends 30% of its engineering time on infrastructure management instead of features. Pick SaaS without thinking through data ownership and you'll face a painful migration when you need custom behaviour the vendor doesn't support.

This decision also shapes your hiring. An IaaS-heavy architecture needs DevOps and SRE engineers. PaaS reduces that need significantly. These downstream consequences are what senior SAs think about before junior SAs even open a browser to compare services.

IaaS — Infrastructure as a Service

🖥️ What you get: Raw virtualised compute (VMs), network, and storage. You manage everything above the hypervisor: OS installation, OS patching, runtime (Node/Python/Java), middleware, and your application.

AWS: EC2  |  Azure: Virtual Machines  |  GCP: Compute Engine
When to choose IaaS: You're lifting and shifting a legacy application that has specific OS or kernel dependencies. You need a GPU instance for machine learning or video encoding. You need a custom network topology. You have an existing 24/7 workload that justifies dedicated infrastructure. You need Windows Server with specific configurations.
When NOT to choose IaaS: You're building a new greenfield web API. Your team has no DevOps capability. You're a startup that needs to move fast. You're trying to prove a concept. IaaS overhead will slow you down dramatically.
📖 Real Example — The Lift-and-Shift
A retailer has a 15-year-old ERP system running on Windows Server 2016 with a custom SQL Server configuration. It can't be containerised — it's tightly coupled to the OS, uses registry keys, and requires specific .NET Framework versions. This is a textbook IaaS use case. They lift the VMs to Azure Virtual Machines unchanged. The system runs exactly as before. Then — over 18 months — they modernise it piece by piece. IaaS was the right starting point because it unblocked the migration without requiring a full rewrite.

PaaS — Platform as a Service

🏗️ What you get: A managed runtime environment. You deploy your application code; the platform handles OS, patching, runtime versioning, load balancing, and basic scaling. You manage your app and data only.

AWS: Elastic Beanstalk, App Runner  |  Azure: App Service  |  GCP: App Engine  |  Others: Heroku, Render, Railway
When to choose PaaS: You're building a web application or API. Your team is primarily developers, not infrastructure engineers. You want to deploy in days not months. You need built-in auto-scaling without managing it yourself. You're doing a PoC or MVP. Your code can run on a standard runtime (Node 18, Python 3.11, Java 17).
📖 Real Example — The Fintech Startup
A 6-person fintech startup is building a loan origination API. They need to launch in 8 weeks, have no DevOps hire, and need to pass a security audit. They choose Azure App Service (PaaS). Deployment is a git push. Auto-scaling handles their variable traffic. Microsoft manages OS patching. Their 6 engineers spend 100% of their time on business logic. 8 weeks later they launch on time. Had they chosen IaaS, they'd have spent weeks configuring VMs, load balancers, and security groups — and probably missed the launch date.

SaaS — Software as a Service

☁️ What you get: A fully managed, fully operated application. You configure it and use it. You own nothing except your data and how you've configured the product.

Examples: Salesforce (CRM), Google Workspace (email/docs), Slack (comms), Datadog (monitoring), GitHub (version control), Snowflake (data warehouse), Auth0 (authentication)
When to choose SaaS: The capability is not your core business differentiator. The problem is solved commodity software. You want zero operational overhead for this capability. You can accept the vendor's data residency and security posture. The integration APIs are sufficient for your needs.
The SaaS trap — vendor lock-in: Three years after adopting a SaaS CRM, a company needs to export all their customer interaction history to feed a new AI model. The vendor charges £50,000 for a bulk data export and the format is proprietary. Always negotiate data portability rights upfront, and understand the export formats before you commit.
🎯 The Build vs Buy Rule: "If it's not your core differentiator, buy it." Authentication, email delivery, monitoring, CRM — these are solved problems. Building them yourself is a distraction. Your competitive advantage is your core product, not the authentication layer that wraps it.

FaaS — Function as a Service (Serverless)

⚡ What you get: You write individual functions that are triggered by events. The platform handles everything: scaling from zero to millions of invocations, infrastructure, patching, runtime. You pay per execution (typically £0.0000002 per invocation).

AWS: Lambda  |  Azure: Functions  |  GCP: Cloud Functions
When to choose FaaS: Event-driven processing (file uploaded → process it). Irregular or unpredictable traffic patterns. Scheduled jobs (cron). Lightweight API backends. Glue code between services. Anywhere you'd previously have written a script on a server.
FaaS limitations to know: Cold starts add latency (50ms–3s depending on runtime and configuration). Maximum execution time of 15 minutes on Lambda. Not suitable for long-running processes. Stateless by nature — can't hold connections. Debugging and local development are harder. Can get expensive at very high sustained throughput compared to a reserved container.
📖 Real Example — The Document Processing Pipeline
An insurance company receives 50,000 PDF policy documents per month for processing — but they arrive in batches, often thousands in a few hours followed by days of nothing. With EC2, they'd have to keep enough servers running to handle peak load 24/7. With Lambda: when a PDF is uploaded to S3, an event triggers a Lambda function. Lambda auto-scales to process 1,000 PDFs simultaneously. When the batch is done, it scales back to zero. Cost before: £1,200/month (always-on EC2). Cost after: £38/month (Lambda, pay per invocation). This is the killer use case for FaaS.

The Shared Responsibility Model

As you move from IaaS → PaaS → SaaS → FaaS, you give up control and gain simplicity. The cloud provider takes on more of the security and operational burden. But you always retain responsibility for:

Your permanent responsibilities regardless of model:
Identity and Access Management (who can access what) · Your application's logic and business rules · Your data (classification, encryption keys, retention policies) · Correctly configuring the services you consume · Compliance with regulations that apply to your data

A common misconception: "We're on SaaS/PaaS so security is the vendor's problem." It isn't. The vendor secures the platform. You secure how you use it — the permissions, the data you put in it, the API keys you generate, and the integrations you build.

⚙️ Phase 2 · Cloud Foundations
Compute: VMs, Containers & Serverless
The three compute paradigms — and how to pick the right one every time.

The Three Compute Paradigms

Every workload you'll ever architect runs on one of these three models — or a combination. The right choice dramatically affects your operational burden, cost profile, scaling behaviour, and the team skills you need. Getting this wrong doesn't just cost money — it shapes the engineering culture of the team for years.

Virtual Machines (EC2, Azure VMs, GCP Compute)

🖥️ How it works: A hypervisor (Xen, Hyper-V, KVM) virtualises physical hardware. You get a slice of a physical server that behaves like a dedicated machine. You choose the OS, install software, configure networking. Boot time: 1–3 minutes. Persistence: runs until you stop it. Billing: by the second (AWS) or minute, even when idle.
Best for: Long-running stateful services. Custom OS/kernel requirements. GPU workloads (ML training, video rendering). Lift-and-shift migrations. High-performance databases that need direct disk access. Workloads with predictable, sustained load where Reserved Instances give 60–72% cost savings.
Watch out for: You're responsible for OS patching — miss a critical CVE and you're exposed. Vertical scaling (making the VM bigger) requires a reboot and brief downtime. Idle VMs still cost money — teams forget to stop dev/test environments. Snowflake servers: VMs that have been manually tweaked so many times nobody knows what's on them.
📖 Real Pattern — Graviton Instances
AWS Graviton3 (ARM-based) instances offer 40% better price-performance than equivalent x86 instances for most workloads. Netflix migrated a significant portion of their EC2 fleet to Graviton and reported 40% cost savings with equal or better performance. If you're recommending EC2 in 2024, you should be recommending Graviton unless there's a specific reason not to (x86-only binary dependencies, for example). Mentioning this in an interview signals you're current.

Containers (Docker, ECS, EKS, GKE, AKS)

🐳 How it works: Containers package your application code + runtime + dependencies into a portable image. Multiple containers share the host OS kernel but are isolated from each other. No OS to manage per container. Boot time: seconds. Images are immutable — the same image runs in dev, staging, and production. This eliminates "works on my machine" problems.
Best for: Microservices architectures. Teams wanting consistent dev/prod parity. Applications that need to scale independently (scale just the checkout service, not the whole app). Multi-cloud portability. CI/CD pipelines (build once, deploy anywhere).
Container orchestration with Kubernetes (K8s): When you have many containers, you need something to manage them — scheduling them onto nodes, restarting failed containers, managing network routing, rolling deployments. Kubernetes does all this. EKS (AWS), GKE (Google), and AKS (Azure) are managed Kubernetes services — they handle the control plane so you only manage the worker nodes and your workloads. For most teams, a managed K8s service is the right choice over self-managed.
📖 Real Example — Spotify's Container Journey
Spotify started with ~150 manually managed VMs per service. Deployments took 30+ minutes and required coordination between teams. After containerising with Docker and moving to Kubernetes, they reduced deployment time to under 2 minutes, improved resource utilisation by 3x (fewer machines doing more work), and gave individual teams full ownership of their service lifecycle. The shift from VMs to containers was one of the biggest engineering productivity improvements in their history.

Serverless / FaaS (Lambda, Cloud Functions, Azure Functions)

⚡ How it works: You deploy a function (a piece of code). The platform spins up an execution environment when an event triggers it, runs your code, and tears it down. You never see a server. Scaling is automatic and instantaneous. You pay only for actual execution time, measured in milliseconds.
Best for: Event-driven processing (S3 upload → resize image, SQS message → process order). API backends with variable or unpredictable traffic. Scheduled jobs. Glue code and automation. Anything where you'd previously have cron'd a script on a server. Startups that want zero idle cost.
Cold start problem — explained properly: When no execution environment exists for your function, AWS must initialise one. For a Node.js function this takes ~200ms. For Java with Spring Boot it can take 3–5 seconds. This "cold start" latency only happens when there's been no traffic for a while. Solutions: Provisioned Concurrency (Lambda keeps warm instances ready, but you pay for them), lighter runtimes (use Node/Python/Go instead of Java for latency-sensitive functions), or SnapStart (AWS's JVM optimisation feature).
📖 Real Cost Comparison — Lambda vs Always-On EC2
A webhook processor receives 2 million events per month. Each takes ~100ms to process.

EC2 t3.small (always-on): ~£12/month regardless of load
Lambda: 2M × 100ms × 128MB RAM = ~£0.34/month

Lambda is 35x cheaper here. But flip the scenario: 100M events/month with sustained high traffic — Lambda becomes £17/month while a reserved EC2 instance handling the load is £8/month. The crossover point matters. Know your traffic pattern before you commit.

The Decision Framework

🧭 How to choose in an interview or real project:

1. Is it event-driven and short-lived (<15 min)? → Serverless (Lambda)
2. Does it need to run continuously or hold state? → VM or Container
3. Does it need to scale independently from other services? → Container
4. Does it have specific OS/kernel/GPU requirements? → VM
5. Is your team small with no DevOps? → Fargate (managed containers, no node management) or Lambda

Most modern architectures mix all three: Lambda for async triggers, containers (Fargate/ECS) for core APIs, VMs for databases or GPU workloads.

🗄️ Phase 2 · Cloud Foundations
Storage: Object, Block & File
Three fundamentally different storage types — knowing which to use and why is a core SA skill.

Why Storage Choice is More Consequential Than It Looks

Storage decisions are some of the hardest to undo. Once you've built your application to write to block storage, migrating to object storage is a significant engineering effort. Once you've stored petabytes in one format, moving it is expensive and time-consuming. Getting this right at design time is critical.

Storage also has major cost implications. Object storage (S3) costs £0.023/GB/month. Block storage (EBS gp3) costs £0.088/GB/month — nearly 4x more. File storage (EFS) costs £0.30/GB/month — 13x more than S3. Choosing block storage for data that should live in S3 can mean paying 4x more for the same bytes.

Object Storage — S3, Azure Blob, GCS

🪣 How it works: Files are stored as objects — each has a unique key (essentially a path), the data itself, and metadata. There's no file system hierarchy — it's a flat namespace. You access objects via HTTP API (PUT to write, GET to read). Infinitely scalable — S3 can store exabytes. Durability: 99.999999999% (11 nines) — meaning if you store 10 million objects, you can expect to lose one every 10,000 years.
Use for: User-uploaded files (images, videos, documents). Static website hosting. Application backups and disaster recovery. Data lake raw/processed layers. Machine learning training datasets. Log archives. Distribution of software or media via CDN. Anything accessed via URL.
S3 Storage Classes — know these for interviews:
S3 Standard — frequent access, highest cost. S3 Standard-IA — infrequent access, 40% cheaper but retrieval fee. S3 Glacier Instant Retrieval — archive with millisecond retrieval. S3 Glacier Deep Archive — cheapest (£0.001/GB) but 12-hour retrieval. S3 Intelligent-Tiering — automatically moves data between tiers based on access patterns. Mentioning tiering classes in a cost optimisation discussion always impresses interviewers.
📖 Real Example — Netflix and S3
Netflix stores the master copies of every piece of content they've ever licensed or produced in S3 — hundreds of petabytes. They transcode each title into 1,200+ different encoding variants (different resolutions, codecs, bitrates for different devices and network conditions) and store all variants in S3. CloudFront CDN serves them globally. The entire content delivery pipeline is built on object storage. There is no "file server" in the traditional sense. This is object storage at its most powerful.

Block Storage — EBS, Azure Managed Disks, GCP Persistent Disks

💾 How it works: A raw storage volume attached to a single VM — behaves exactly like a physical hard drive. The OS formats it with a filesystem (ext4, NTFS) and mounts it. Sub-millisecond latency. High IOPS (EBS gp3 gives you 3,000 IOPS baseline, up to 16,000). Can only be attached to one EC2 instance at a time (with some exceptions for EBS Multi-Attach).
Use for: Operating system volumes for EC2 instances. Relational databases (PostgreSQL, MySQL, Oracle) — they need low-latency disk access for write-ahead logs and data files. Any application that needs a traditional mounted filesystem. High-frequency transaction processing where latency matters.
Common mistake — orphaned volumes: When an EC2 instance is terminated, its attached EBS volumes are not automatically deleted by default. They remain — and you keep paying for them. A common FinOps finding is dozens of EBS volumes paying £0.088/GB/month for data nobody is using. Always set the DeleteOnTermination flag unless you specifically need the data to survive instance termination.

File Storage — EFS, Azure Files, FSx

📁 How it works: A Network File System (NFS) that multiple compute instances can mount simultaneously. Unlike block storage (one instance only), file storage can be attached to hundreds of containers or VMs at the same time. All instances see the same files in real time. Managed by the cloud provider — you never manage the underlying disk.
Use for: Shared content repositories accessed by multiple servers. CMS media libraries where multiple web servers need to read/write the same image files. HPC (High Performance Computing) clusters where compute nodes share a scratch workspace. Lift-and-shift of applications that expect a shared network drive. Container clusters that need a shared persistent volume.
📖 Real Example — The Media Processing Cluster
A broadcast company runs a cluster of 50 video processing workers. Each worker picks up a job, reads the source video, processes it, and writes the output. All 50 workers need to access the same input and output directories. With block storage (EBS), each worker would need its own copy of the video — 50x storage cost and the need to sync files between workers. With EFS, all 50 workers mount the same filesystem. One copy of every file, accessible to all workers simultaneously. This is the canonical file storage use case.

The Decision Framework + Real Architecture Example

🧭 Quick Decision Rules:
Multiple instances need to read/write simultaneously? → File (EFS)
One instance needs a fast local disk (database, OS)? → Block (EBS)
Storing files accessed via API, URL, or CDN? → Object (S3)
Archiving data you rarely access? → Object with Glacier tier
Sharing files between containers in the same cluster? → EFS or a shared object store
Real architecture using all three:
A media platform uses: S3 for user-uploaded videos and processed outputs (cheap, durable, CDN-integrated) · EBS gp3 for the PostgreSQL database that stores user accounts and video metadata (low latency, high IOPS for transactional data) · EFS for the shared rendering workspace used by a cluster of 20 video processing workers (all nodes read/write the same working files simultaneously). Three storage types, three different jobs — each perfectly matched.

🔑 Phase 2 · Cloud Foundations
Identity & Access Management (IAM)
The foundation of cloud security. Get this wrong and nothing else matters.

Why IAM is the Highest-Leverage Security Control

The majority of significant cloud security breaches trace back to IAM failures — not sophisticated exploits, but basic mistakes: credentials hardcoded in GitHub repos, overly broad permissions that should have been scoped, service accounts with admin access "just in case," root account used for day-to-day work. IAM is not glamorous, but it is the single highest-impact security control in your architecture.

The 2019 Capital One breach (100 million customer records) was caused by two IAM failures: a misconfigured WAF that allowed SSRF, and an EC2 metadata role with permissions far broader than needed. If the role had only had access to the specific S3 bucket it needed — nothing else — the breach would have been contained. Least privilege would have saved them £80M in fines and immeasurable reputational damage.

The Principal Hierarchy — Who Can Do What

Users (IAM Users): Long-lived credentials (access key + secret) tied to a specific human or application. The access key is like a username/password that never expires unless rotated. Avoid for services and automation — if the key leaks (e.g., accidentally pushed to GitHub), it's valid until manually revoked. Use for: individual humans who need programmatic access when federation isn't yet set up.
Roles (IAM Roles): Temporary credentials, automatically rotated, assumed by services or humans. An EC2 instance with an IAM Role gets credentials that automatically expire every hour and are refreshed by the AWS metadata service. A Lambda function assumes its execution role for the duration of the invocation. Always prefer roles over users for services. There are no long-lived credentials to leak.
Groups: A collection of users. Attach permissions to the group, not to individual users. When someone joins the DevOps team, add them to the DevOps group — they inherit all permissions. When they leave, remove them from the group — all permissions gone. Simpler to manage at scale than per-user policies.
Service Control Policies (SCPs): Organisation-level guardrails that apply to all accounts in your AWS Organisation. If you have 50 AWS accounts, an SCP can say "no account in this org is ever allowed to disable CloudTrail" or "no account can create resources outside eu-west-1 and eu-west-2." SCPs are like circuit breakers — even if an account's admin tries to violate the policy, they can't. This is advanced IAM but it's what enterprise architects implement.

Policies — The Permission Language

IAM Policies are JSON documents that define what actions are allowed or denied on what resources. Every API call in AWS is controlled by a policy evaluation. Understanding how policies evaluate is essential for secure architecture.

Policy evaluation logic (memorise this):
1. Start with implicit Deny — everything is denied by default
2. Any explicit Allow grants access
3. Any explicit Deny always wins — even over an Allow

This means: if you have an Allow policy for S3:* and a Deny policy for S3:DeleteObject, the Deny wins. You can write to S3 but never delete. Explicit Deny is your safety net for dangerous operations.
Wildcard permissions — the danger zone:
"Action": "*" means the role can call any AWS API.
"Resource": "*" means the permission applies to every resource in your account.

Combining both ("Action": "*", "Resource": "*") is essentially root access. This pattern appears in "quick fix" policies all the time and is a critical finding in any security review. Always scope to the minimum Action and specific Resource ARNs.
📖 Real Policy — Lambda to Specific S3 Bucket
Bad policy (wildcard): grants Lambda access to all S3 buckets in your account. One Lambda bug could expose all your data.

Good policy: Action: ["s3:GetObject", "s3:PutObject"], Resource: "arn:aws:s3:::my-uploads-bucket/*". This Lambda can only read/write to one specific bucket. If it's compromised, the blast radius is limited to that one bucket, not your entire S3 estate.

Federation — The Enterprise Standard

Large organisations don't create individual IAM Users for every employee. They use their existing Identity Provider (IdP) — typically Active Directory, Okta, or Azure AD — and federate access to AWS. This means:

SAML 2.0 / OIDC Federation: Employee logs into Okta with their company credentials. Okta asserts their identity to AWS. AWS issues temporary credentials valid for the session. No IAM User exists — the human's identity lives in the IdP. When an employee leaves, disabling their Okta account immediately revokes all AWS access. Central governance, no stale credentials.
AWS IAM Identity Centre (formerly SSO): AWS's managed federation service. Integrates with Active Directory or external IdPs. Gives employees a single portal to access multiple AWS accounts with appropriate permissions. Logs all access. This is the recommended approach for any organisation with more than a handful of AWS accounts.

🎯 Senior SA Interview Answer on IAM: "I'd enforce least privilege using IAM Roles for all service-to-service access — never long-lived access keys. I'd federate human access through our IdP via IAM Identity Centre, so employee lifecycle management is centralised. I'd use SCPs at the organisation level to enforce guardrails like CloudTrail being always-on and resource creation locked to approved regions. What is a Solutions Architect?
The mindset, responsibilities, and superpowers of your role.

The Role in One Sentence

A Solutions Architect is the translator between what the business needs and what technology can deliver — making complex technical decisions that have real commercial consequences.

🏗️ Real World Analogy
Think of yourself as an architect designing a skyscraper. You don't lay every brick yourself — but you make the critical decisions: structural load, materials, safety systems, and aesthetics. The builders (engineers) execute your plan. If your plan is wrong, the building falls. If it's brilliant, it stands for 50 years. An architect who just draws pretty pictures without understanding load-bearing physics isn't an architect — they're a liability. Same applies to you.

In practice, this means you will routinely sit in a room where the CTO wants to move fast, the security team wants to slow down, the CFO wants to cut costs, and the engineering team wants to use the latest technology. Your job is to propose an architecture that threads that needle — and explain why in terms each person understands.

What Do You Actually Do Day to Day?

This varies by company, but a typical week as an SA at a mid-size tech company looks like this:

Monday: Requirements workshop with a product team building a new payments feature. You ask probing questions: How many transactions per second at peak? What's our SLA? Does this touch PCI-DSS data? You leave with a list of constraints that will shape every decision.
Tuesday–Wednesday: You sketch out 3 candidate architectures — one cheap and fast to build, one that scales to 10x current load, one that a small team can actually operate. You document the trade-offs of each in an Architecture Decision Record (ADR).
Thursday: Architecture review. You present your preferred option to a mix of engineers, a product manager, and a security lead. You defend your choices under questioning. The security lead raises a concern about token storage — you hadn't considered that angle. You revise.
Friday: You update the ADR with the final decision and its rationale. You write a one-pager for the CTO. You pair with the lead engineer to make sure they understand the design intent before they start coding.

Notice: you didn't write much code. You wrote documents, asked questions, drew diagrams, and made decisions. Communication is 60% of the job.

Junior vs Senior Mindset — With Real Examples

Junior SA thinks: "What's the best technology for this?"
Senior SA thinks: "What's the best outcome for the business — and which technology gets us there with acceptable risk and cost?"

This isn't abstract. Here's the same scenario handled two ways:

❌ Junior response to "We need real-time notifications":

"Let's use WebSockets — they're the fastest and most modern option. I'll set up a WebSocket server on EC2."

Problem: WebSockets require persistent connections, meaning you can't easily load balance them. The EC2 instance becomes a single point of failure. When you need to scale, you'll have to re-architect everything.
✅ Senior response to "We need real-time notifications":

"Before I recommend a solution — how many concurrent users do we expect? Are these notifications one-directional (server → client) or bidirectional? Do users need to receive notifications when the app is closed? What's our existing stack?"

After hearing the answers: "For mostly server-to-client, Server-Sent Events (SSE) is simpler and more scalable than WebSockets. If users need notifications when the app is closed, we need push notifications via FCM/APNs. If it's truly bidirectional with high frequency, then WebSockets with a managed service like AWS API Gateway is the right call — not self-managed."

The senior SA gave a better answer because they asked better questions first. That is the whole game.

🎯 Interview Gold: When asked a design question, always start by asking about business requirements, scale expectations, budget constraints, and acceptable downtime. Say out loud: "Before I dive in, I have a few clarifying questions." Interviewers actively look for this — it's how they tell apart people who've read about architecture versus people who've practised it.

Architecture Decision Records (ADRs) — Your Paper Trail

One of the most underrated skills of a senior SA is documenting why decisions were made, not just what was decided. An ADR is a short document (usually 1–2 pages) that captures:

Status: Proposed / Accepted / Deprecated
Context: What situation forced this decision?
Decision: What did we choose?
Consequences: What are the trade-offs we accepted?

Two years from now, when a new engineer asks "why on earth did we build it this way?" — the ADR has the answer. Without it, institutional knowledge dies when people leave. Senior SAs write ADRs for every significant decision. It also protects you: if the decision turns out to be wrong, the ADR shows you made a reasonable choice given the information available at the time.

The SA's Non-Functional Requirements (NFRs)

These are the invisible requirements that define your architecture quality. Unlike functional requirements ("the user can reset their password"), NFRs cut across the entire system. Missing them at design time is extremely expensive to fix later.

ScalabilityAvailabilityReliabilityPerformanceSecurityMaintainabilityObservabilityCost-efficiencyCompliancePortability
Real example of an NFR being missed: A startup builds a healthcare app. They focus entirely on features. Nobody specifies that patient data must be encrypted at rest and in transit (a compliance NFR). Six months in, they're in talks with NHS. NHS requires DSPT compliance. The team discovers half their database fields are stored unencrypted. Re-architecting this costs 3 months of engineering time and delays the NHS deal by a quarter. A single conversation about NFRs at the start would have prevented this.

Real War Story: When SA Thinking Saves a Project

📖 Case Study — E-commerce Flash Sale
An e-commerce company is planning a Black Friday flash sale. The marketing team tells engineering: "We're expecting 10x our normal traffic for 4 hours." A junior engineer proposes upgrading all servers to the next size up — vertical scaling.

The SA stops the meeting and asks five questions:
1. What's our current peak, and what's the predicted peak in absolute numbers?
2. Which parts of the system are actually the bottleneck — web servers, database, or third-party payment API?
3. What's the cost of a 1-hour outage during the sale vs. the cost of over-provisioning?
4. Do we need to scale before the sale or can we use auto-scaling during it?
5. What's our rollback plan if the new infrastructure behaves unexpectedly?

The answers reveal: the database is the real bottleneck, not the web servers. The SA recommends adding read replicas and a Redis caching layer in front of the product catalog — the most-read data. Total cost: £200/month extra for 2 weeks of testing plus the sale period. The vertical scaling plan would have cost £4,000 and wouldn't have solved the actual problem. The sale goes flawlessly.

This is what being a Solutions Architect actually looks like in practice. It's not about knowing every technology — it's about asking the right questions and connecting the answers to the right solution.

The Well-Architected Framework
The 6 pillars that define great architecture. A senior interview staple.

Why This Matters

AWS's Well-Architected Framework is the closest thing to a universal standard for evaluating cloud architectures. Even if you work on Azure or GCP, the principles translate directly. It gives you a shared vocabulary with every cloud-literate engineer and interviewer on the planet.

More practically: when you join a new company and need to assess an existing system, this framework is your checklist. When you're designing something new, it's your quality gate. When you're in an interview and they ask "how would you evaluate this architecture?" — this is your answer structure.

🎯 Pro Tip: In any system design interview, say "Let me walk through this from a Well-Architected perspective..." It immediately signals structured, mature thinking. Most candidates jump straight to tech choices. You'll stand out by thinking in frameworks.

Pillar 1 — 🔧 Operational Excellence

The ability to run and monitor systems effectively, and to continually improve supporting processes and procedures. This is about how you operate, not just what you build.

Key practices: Infrastructure as Code (IaC) using Terraform or CloudFormation — so your infrastructure is reproducible and version-controlled. CI/CD pipelines so deployments are automated and consistent. Runbooks for every common incident type. Blameless post-mortems after every outage so you learn rather than blame. Small, frequent deployments rather than large, risky ones.
📖 Real Example — The £2M Deployment Mistake
A financial services firm deployed a monolith quarterly. Each release was months of work bundled into one giant deployment. When something broke, it was nearly impossible to isolate which change caused it. One quarter a deployment corrupted trading data — it took 4 days to identify the cause. Total cost: £2M in lost trades and remediation. After moving to small daily deployments via a proper CI/CD pipeline, their mean time to detect (MTTD) dropped from days to minutes. Operational Excellence isn't boring — it's directly tied to revenue.
🎯 Interview Answer: "For operational excellence I'd implement IaC from day one — probably Terraform — so we can version control our infrastructure. CI/CD via GitHub Actions or Jenkins. Structured logging and distributed tracing so we can actually debug production issues. And a post-mortem culture so every incident generates a concrete improvement."

Pillar 2 — 🛡️ Security

Protecting information, systems, and assets while delivering business value. Security is not a feature you add at the end — it's a design constraint from day one. The principle of defence in depth means you layer multiple security controls so that no single failure exposes you.

Key practices: Least privilege IAM (nobody gets more permissions than they need). Encryption at rest and in transit — always, no exceptions. Automated security scanning in your CI pipeline (SAST, dependency scanning). Secrets management via AWS Secrets Manager or HashiCorp Vault — never hardcode credentials. WAF in front of public-facing services. VPC with private subnets for anything that doesn't need internet access.
📖 Real Example — The Capital One Breach (2019)
A misconfigured WAF allowed an attacker to exploit a Server-Side Request Forgery (SSRF) vulnerability. The attacker hit the EC2 metadata endpoint and retrieved IAM credentials. Those credentials had overly broad permissions — violating least privilege. 100 million customer records were exposed. Fine: $80M. Root causes: two failures of this pillar — WAF misconfiguration and over-privileged IAM roles. Had least privilege been enforced, the stolen credentials would have had almost no useful access.
🎯 Interview Answer: "Security is non-negotiable from day one. I'd enforce least-privilege IAM with permission boundaries, deploy behind a WAF, encrypt everything in transit with TLS 1.2+ and at rest with KMS-managed keys, manage secrets via Secrets Manager, and bake security scanning into the CI pipeline so vulnerabilities are caught before they reach production."

Pillar 3 — 🔄 Reliability

The ability of a system to recover from failures and continue to function. The key insight here is design for failure — assume everything will fail, and build so the system heals itself. In AWS, the unit of resilience is the Availability Zone (AZ). A truly reliable system survives the loss of an entire AZ without user-visible impact.

Key practices: Multi-AZ deployments for databases and application tiers. Auto-scaling groups that replace failed instances automatically. Circuit breakers to stop cascading failures when a downstream service is struggling. Retries with exponential backoff and jitter (not just naive retries which can create thundering herds). Health checks and automated failover. Regular DR testing — if you've never tested your failover, you don't actually have one.
📖 Real Example — AWS us-east-1 Outage
In December 2021, AWS us-east-1 had a major outage. Companies that had deployed multi-region or at least multi-AZ with proper failover stayed online. Companies that had assumed a single region would "just work" went down — including some major services that took client-facing dashboards offline for hours. The lesson: reliability requires you to actively design for failure, not passively hope it won't happen.
🎯 Interview Answer: "I'd design for failure from the start: multi-AZ for all stateful components, circuit breakers between services, health checks with automatic replacement of failed instances, and regular chaos engineering — deliberately killing instances in staging to verify recovery procedures actually work."

Pillar 4 — ⚡ Performance Efficiency

Using computing resources efficiently to meet system requirements, and maintaining that efficiency as demand changes. This is not just about speed — it's about using the right resources for the job.

Key practices: Choose the right instance type for the workload — compute-optimised (C-series) for CPU-heavy tasks, memory-optimised (R-series) for in-memory databases, storage-optimised (I-series) for high I/O. Use caching aggressively: CDN for static content, ElastiCache/Redis for hot database queries, in-process caching for repeated computations. Scale horizontally rather than vertically where possible — it's more resilient. Profile before optimising — don't guess where bottlenecks are.
📖 Real Example — The Database That Didn't Need to Scale
A SaaS company was spending £8,000/month on a massive RDS instance because their app was slow. A consultant profiled the application and found 80% of the load was 3 specific queries hitting a product catalog — the same 200 products read thousands of times per minute. Adding a Redis cache in front of those 3 queries took 2 days of engineering time. Database CPU dropped from 85% to 12%. They downgraded to a £900/month instance. Performance improvement: 40% faster response times. Cost saving: £7,100/month. This is Performance Efficiency in action.

Pillar 5 — 💰 Cost Optimization

Running systems at the lowest price point while still delivering business value. Cost is a non-functional requirement like any other — ignoring it creates technical debt of a financial kind.

Key practices: Right-size instances — the default choice is often too large. Use Reserved Instances or Savings Plans for predictable workloads (up to 72% savings vs On-Demand). Spot Instances for fault-tolerant batch jobs (up to 90% savings). Tag every resource for cost attribution — you can't optimise what you can't measure. Review your bill monthly. Use S3 Intelligent-Tiering for data you access infrequently. Delete unattached EBS volumes, idle NAT gateways, and unused Elastic IPs.
📖 Real Example — The Zombie Resources
A startup conducted their first cloud cost review 18 months after launch. They found: 12 EBS volumes attached to terminated instances (£180/month), a NAT gateway in a region they'd stopped using (£90/month), 3 RDS snapshots being retained indefinitely from a database they'd migrated away from (£240/month), and a dev environment running 24/7 that was only used 8 hours a day (could be scheduled to stop at night, saving £380/month). Total wasted spend found in one afternoon: £890/month. £10,680/year. FinOps is not glamorous but it's very valuable.

Pillar 6 — 🌱 Sustainability

Minimising the environmental impact of running cloud workloads. This is the newest pillar (added 2021) and is increasingly asked about in interviews, especially at companies with ESG commitments or public net-zero targets.

Key practices: Choose AWS regions powered by renewable energy (eu-west-1 Ireland, eu-central-1 Frankfurt have strong green credentials). Right-size to reduce waste — an idle oversized instance wastes energy. Use managed services (Lambda, Fargate) which have higher utilisation density than dedicated EC2. Consolidate workloads. Delete what you're not using. Graviton (ARM) instances use ~60% less energy than equivalent x86 for the same workload.
🎯 Interview Angle: Most candidates don't mention sustainability at all. Bringing it up proactively — even briefly — signals awareness of the broader picture and is especially impressive at companies with public ESG commitments. Say: "I'd also consider the sustainability pillar — choosing the eu-west-1 region for its renewable energy mix and using Graviton instances where possible for the energy efficiency gains."

Running a Well-Architected Review (WAR)

A WAR is a structured conversation you run with a team — typically using AWS's questionnaire tool in the console — that produces a report of High Risk Issues (HRIs) and Medium Risk Issues (MRIs). You're not looking for perfection — you're looking for the most dangerous gaps.

Typical WAR process:
1. Schedule 2-hour session with the team who built and operates the system
2. Work through the pillar questions in the AWS WAR Tool (or equivalent)
3. Each question asks "do you do X?" — be honest, not aspirational
4. Tool produces a risk report with severity ratings
5. You produce a prioritised remediation plan — fix HRIs first
6. Re-review in 6 months
🧭 Mental Model
Think of the 6 pillars as 6 lenses. Every architecture decision you make, look through each lens: "Does this choice make me more or less operationally excellent? More or less secure?" You won't always optimise all 6 simultaneously — that's where trade-offs come in. A startup in week 2 legitimately prioritises speed over reliability. A bank legally cannot. Context always drives weighting.

🎭 Lesson 3 · Architecture Fundamentals
The Art of Trade-offs & CAP Theorem
The concept that separates junior from senior thinking.

There Is No Perfect Architecture

This is the most important sentence in architecture. Every design decision is a trade-off. The junior SA looks for the "right answer." The senior SA asks "right for whom, when, and at what cost?" Your entire career progression is learning to make better trade-off decisions faster, with more confidence, and to articulate your reasoning clearly to others.

🧭 The Triangle You Can't Escape
Engineering has a classic constraint: Fast, Cheap, Good — pick two. Architecture has its own version. For any system, you're constantly balancing: speed to market, reliability, cost, security, and developer experience. Squeezing one almost always expands another. Recognising these tensions early is what separates senior thinking.

CAP Theorem — The Foundation

Proposed by Eric Brewer in 2000 and formally proven in 2002, CAP Theorem states that in any distributed data system, you can only guarantee TWO of these three properties simultaneously:

🔒
Consistency
Every read returns the most recent write — or an error. No stale data ever.
⚡
Availability
Every request gets a response — though it might not contain the latest data.
🌐
Partition Tolerance
System keeps operating even if some nodes can't communicate with others.
The critical insight: Network partitions are not optional in distributed systems — they will happen. Hardware fails, cables get cut, DNS goes wrong. So Partition Tolerance is not really a choice — it's a requirement. That means you're actually choosing between CP (sacrifice availability when partitions occur) and AP (sacrifice consistency when partitions occur). This is a business decision disguised as a technical one.

CP vs AP — Real Database Examples

CP Databases (Consistent + Partition Tolerant):
PostgreSQL, MySQL with synchronous replication, MongoDB in strong consistency mode, Apache HBase, DynamoDB in strong consistency mode.

When a partition occurs, these systems will refuse to serve reads rather than return stale data. Better to say "service unavailable" than to serve wrong data.
AP Databases (Available + Partition Tolerant):
Apache Cassandra, CouchDB, DynamoDB in eventual consistency mode, Riak, DNS itself.

When a partition occurs, these systems keep responding — but different nodes may return different values until they sync back up. Eventually consistent.

When to Choose CP vs AP — With Real Examples

🏦 Choose CP — Banking & Payments:
Imagine two people with access to a joint bank account, each on a different continent. A network partition occurs between the two database nodes. An AP database might allow both to withdraw the last £500 simultaneously — the account is now overdrawn by £500. A CP database refuses one of those transactions until the partition resolves. The cost: a brief service disruption. The alternative cost: financial loss and regulatory breach. Banks always choose CP.
🛒 Choose AP — Shopping Cart & Product Catalog:
Amazon's Dynamo paper (which inspired DynamoDB and Cassandra) specifically describes how Amazon chose AP for their shopping cart. If two clients add items to a cart during a partition, the worst case is they see slightly different versions temporarily. After the partition resolves, the system merges the carts. The alternative — refusing to let customers add to cart during a network hiccup — would cost millions in abandoned purchases. For product catalogs, showing a price that's 2 seconds stale is infinitely better than showing an error.
🏥 Choose CP — Healthcare Dosage Tracking:
A nurse checks a patient's last medication dose before administering another. If the system returns stale data due to a partition (AP behaviour), the patient might receive a double dose. This is a life-safety scenario — the system must be CP. Return an error and have the nurse check the physical chart rather than risk incorrect data. The inconvenience of an error message is always preferable to patient harm.
📱 Choose AP — Social Media Feeds:
Twitter, Instagram, Facebook — all AP. When you like a post, the like count across all servers might be inconsistent for a few seconds. Some users see 1,042 likes, others see 1,041. Nobody cares. But if Instagram went down every time there was a network hiccup between their data centres, that would be catastrophic. The business value of availability massively outweighs the cost of momentarily stale like counts.

PACELC — The Full Picture CAP Misses

CAP only describes behaviour during a partition. But systems spend most of their time running normally. PACELC (proposed by Daniel Abadi) extends the model:

PACELC stands for:
If there's a Partition, choose between Availability and Consistency (the CAP choice).
Else (normal operation), choose between Latency and Consistency.

To achieve perfect consistency under normal operation, writes must be acknowledged by all replicas before returning success. That takes time — higher latency. If you want lower latency, you acknowledge the write from one node and let replication happen asynchronously. But now there's a window where a read from another node returns stale data.
📖 Real Example — Global Leaderboard
A gaming company has a global leaderboard. Their database has nodes in London, Singapore, and New York. A player in Singapore gets a high score. For strict consistency (PACELC/C), the write must reach London and New York before acknowledging success — adding 200ms of latency. For the leaderboard showing globally accurate data within 5 seconds rather than 200ms, that's a bad trade-off. They choose eventual consistency (PACELC/L) — the score propagates within seconds, latency stays under 20ms, and nobody notices the brief inconsistency. This is PACELC thinking applied.
🎯 Interview Power Move: After explaining CAP, say: "CAP describes what happens during a partition, but PACELC reminds us that even under normal operation, latency and consistency are in tension. A globally distributed system is constantly making this trade-off on every write." This alone puts you in the top 10% of candidates.

Other Critical Trade-offs You'll Face Daily

⚡ Latency vs Throughput: A system can be optimised for low latency (respond fast to each request) or high throughput (process many requests per second). Batch processing optimises for throughput; real-time APIs optimise for latency. You rarely get both for free.
🔧 Build vs Buy: Should you build an authentication system or use Auth0/Cognito? Build a search engine or use Elasticsearch/OpenSearch? The rule of thumb: if it's not your core business differentiator, buy. Every line of code you write is a liability that needs testing, maintaining, and securing. Buying lets your team focus on what makes you unique.
🔒 Monolith vs Microservices: Microservices give you independent deployability and scalability — but add enormous operational complexity (service discovery, distributed tracing, network failure handling). Monoliths are simpler to develop and operate — but harder to scale individual components and can become unmaintainable. Amazon, Netflix, and Uber all started as monoliths and split them later. For most teams under 50 engineers, a well-structured monolith is genuinely the better choice.
💰 Cost vs Performance: The fastest solution is almost never the cheapest. Caching reduces database load but adds memory cost and cache invalidation complexity. Reserved instances are cheaper than On-Demand but require upfront commitment. Auto-scaling handles peaks but the per-unit cost is higher than a reserved baseline. You're always balancing the cost of a solution against the cost of the problem it solves.

Architecture Diagrams & Communication
The C4 Model and how to communicate designs like a senior.

Why Diagrams Are a Superpower

A great architecture diagram can align a 20-person team in 10 minutes. A bad one creates confusion for months. As an SA, your ability to communicate visually is as important as your technical knowledge — in many roles, more so.

Diagrams do three things that words can't: they reveal gaps in your thinking, they expose misalignments between what different people understood, and they create a shared reference point that everyone can point to and discuss. If you can't draw it, you probably don't understand it clearly enough to build it.

The Golden Rule: Different audiences need different levels of abstraction. An executive needs a context diagram. A developer needs a component diagram. Never show the same diagram to both — it will either confuse the executive or bore the developer.

The C4 Model — Your Professional Standard

Created by Simon Brown, C4 gives you a consistent, audience-appropriate way to describe any software system. The name comes from its four levels of zoom. Think of it like Google Maps — you start from space and zoom in.

🌍
L1: Context
The system + its users + external systems. For executives and stakeholders. Should fit on a single slide.
📦
L2: Container
Apps, databases, microservices, file stores. For senior engineers and architects. Shows what's running where.
⚙️
L3: Component
Internal structure of one container. For the team building that specific service.
💻
L4: Code
Classes and functions. Rarely drawn manually — usually auto-generated from code. For devs only.

C4 in Practice — A Real E-Commerce Example

Let's walk through all four levels for an e-commerce platform so you can see exactly what goes in each:

L1 — Context Diagram:
One box in the middle: "E-Commerce Platform". Arrows coming in from: "Customer (web browser)", "Customer (mobile app)". Arrows going out to: "Payment Gateway (Stripe)", "Email Service (SendGrid)", "Warehouse Management System (3rd party)". That's it. No databases. No services. No code. Just: what is this system, who uses it, and what does it talk to?
L2 — Container Diagram:
Now zoom into the "E-Commerce Platform" box. Inside, you show: "React SPA (JavaScript, runs in browser)", "API Gateway (AWS, HTTPS)", "Order Service (Node.js, Docker)", "Product Service (Python, Docker)", "User Service (Go, Docker)", "Orders Database (PostgreSQL, AWS RDS)", "Product Cache (Redis, ElastiCache)", "Message Queue (RabbitMQ)". Arrows show how they communicate: Order Service → Orders DB (SQL), Order Service → Message Queue (AMQP), Product Service → Product Cache (Redis protocol).
L3 — Component Diagram (Order Service only):
Zoom into the Order Service container. Show its internal components: "Order Controller (REST endpoint handler)", "Order Validator (business rules)", "Payment Orchestrator (coordinates with Stripe)", "Order Repository (database access layer)", "Event Publisher (emits events to queue)". Show how they interact internally.
L4 — Code (rarely needed):
UML class diagram of the Order Repository class. Which methods it exposes, what it inherits from. Usually auto-generated by your IDE. Almost never drawn by hand in modern practice.
💡 Key Insight
Notice how each level only adds detail to one box from the previous level. You never jump straight from L1 to L3. This discipline keeps diagrams readable and ensures you're designing top-down, not bottom-up.

Diagramming Tools — What to Use When

draw.io / diagrams.net (free, recommended to learn on): Drag-and-drop, works in browser, saves to Google Drive or locally. AWS, Azure, GCP icon libraries built in. This is what most SAs use for day-to-day work.
Lucidchart (paid, enterprise standard): Real-time collaboration, integrates with Confluence and Jira. If you're at a larger company, this is likely what they use. The interface is similar to draw.io so skills transfer.
Structurizr (C4-native, code-based): You define your architecture as code (DSL or Java/C#/Go libraries) and Structurizr renders the diagrams. The major advantage: diagrams stay in sync with code changes and live in version control. This is the gold standard for keeping diagrams up to date — traditional diagrams rot as code evolves.
Excalidraw (for whiteboards and quick sketches): Intentionally sketchy-looking, which signals "this is exploratory" rather than "this is final". Great for workshops and early-stage thinking. Very fast to use. Many engineers use it for interview prep whiteboards.
🎯 Interview Whiteboard Tip: In system design interviews, always start with a context-level box. Say: "Let me establish the system boundary first — here's the user, here's our system, here are the external dependencies." This signals structured thinking and buys you 90 seconds to think about the harder design questions. Then zoom to L2. You rarely need to go deeper in an interview unless specifically asked.

Common Diagramming Mistakes to Avoid

❌ Mixing abstraction levels: Showing a "User" box alongside a "PostgreSQL index" in the same diagram. Pick a level and stay there. Mixing levels confuses everyone.
❌ Unlabelled arrows: An arrow between two boxes means nothing without a label. What protocol? What data? What direction? "REST/HTTPS", "publishes events", "SQL queries" — always label your arrows.
❌ No legend or key: If you use colours or shapes to mean something, say so. Don't make your audience guess that blue boxes are AWS services and green boxes are third-party services.
❌ The "spaghetti diagram": Showing every system and every connection in one diagram. After about 7–8 boxes it becomes unreadable. If your diagram has more than 8–10 boxes, it needs to be split into multiple diagrams at different zoom levels.

The 3 Audiences — With Script

Knowing which diagram to show is half the battle. Knowing how to talk to it is the other half. Here's how a senior SA would walk through the same system for three different audiences:

To the CFO (L1): "This diagram shows our e-commerce platform at a high level. Customers connect through web and mobile. We process payments through Stripe — that's the orange box. Our warehouse system is on the right — that's a third-party system we integrate with. Everything in the blue box is what our team builds and operates."
To the Product Manager (L2): "When a customer places an order, it hits our API Gateway, goes to the Order Service here, which validates the order and calls Stripe for payment. If payment succeeds, it publishes an event to this queue. The Fulfilment Service picks that up and creates the shipment. The customer gets an email from SendGrid here."
To the Engineering Team (L3): "Inside the Order Service, the Controller accepts POST /orders. It passes to the Validator which checks business rules. Then the Payment Orchestrator calls Stripe's API — note it uses idempotency keys to prevent double charges. On success, the Order Repository writes to Postgres and the Event Publisher sends to RabbitMQ using the outbox pattern to guarantee delivery even if the queue is down."
