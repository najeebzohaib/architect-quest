🎯 Q1 · Architecture Fundamentals
"Walk me through how you'd design a system architecture for a new web application that needs to support 10,000 concurrent users."
💡 Tip: Structure your answer around: Presentation → Application → Data layers. Mention scalability from the start.
Start with a load balancer distributing traffic across multiple web servers (horizontal scaling). Use auto-scaling groups so the system expands under load and contracts to save cost. Separate the application layer from the database. Use a managed relational DB (RDS/Cloud SQL) for transactional data, with read replicas to distribute read load. Add a caching layer (Redis/Memcached) to reduce DB hits for hot data. Use a CDN to serve static assets. Finally — add monitoring (CloudWatch/Datadog) and ensure you have circuit breakers to prevent cascade failures. The key is to make each layer independently scalable.
🎯 Q2 · Trade-offs
"Explain the CAP theorem and describe a scenario where you had to choose between consistency and availability."
💡 Tip: Don't just define it — give a concrete example. E-commerce checkout vs product catalog behave differently.
CAP Theorem states that in a distributed system, you can only guarantee two of three: Consistency (all nodes see the same data), Availability (system always responds), or Partition Tolerance (system works despite network splits). Since network partitions happen in real distributed systems, you're really choosing between CP and AP. Example: For a payment service, I'd choose CP — it's better to refuse a transaction than to process a double charge. For a product catalog or social feed, I'd choose AP — stale data for a few seconds is acceptable; an outage is not. The key interview answer is showing you understand the business context drives the choice.
🎯 Q3 · Well-Architected
"How would you evaluate whether an existing architecture is 'good'? What framework or criteria do you use?"
💡 Tip: Reference the Well-Architected Framework pillars. Show structured thinking, not gut feel.
I use the AWS Well-Architected Framework as a structured lens: Operational Excellence (can we deploy and monitor reliably?), Security (are we protecting data and following least-privilege?), Reliability (does it recover from failure automatically?), Performance Efficiency (are we using resources appropriately for the load?), Cost Optimization (are we paying for what we actually need?), and Sustainability (are we minimizing environmental impact?). I'd run a Well-Architected Review with the team — interviewing stakeholders, reviewing architecture diagrams, checking IaC configs — and produce a findings report with prioritized remediation items. Interviewers love this answer because it shows process, not just instinct.

📦 Phase 2 · Cloud Foundations
IaaS, PaaS, SaaS & FaaS
The four cloud service models — and when each one is the right call.

Why This Matters

Choosing the wrong service model is one of the most expensive mistakes in cloud architecture. Picking IaaS when you need PaaS means months of unnecessary infrastructure management. Picking SaaS when you need control means painful vendor lock-in later.

The Four Models

🖥️ IaaS — Infrastructure as a Service
You get raw compute, storage, and networking. You manage the OS, runtime, middleware, and app. Maximum control, maximum responsibility. Use when you need fine-grained control, custom OS configs, or are lifting-and-shifting legacy workloads.
Examples: AWS EC2, Azure VMs, GCP Compute Engine
🏗️ PaaS — Platform as a Service
The cloud manages OS, runtime, and middleware. You only manage your app and data. Faster to deploy, less ops overhead. Use for web apps, APIs, and microservices where you don't need OS-level control.
Examples: AWS Elastic Beanstalk, Heroku, Google App Engine, Azure App Service
☁️ SaaS — Software as a Service
The entire stack is managed. You just configure and use it. Zero infrastructure concern. Use for commodity tools (email, CRM, HR) where differentiation isn't needed.
Examples: Salesforce, Google Workspace, Slack, Datadog
⚡ FaaS — Function as a Service (Serverless)
You deploy individual functions triggered by events. No servers to manage, pay only per invocation. Use for event-driven workloads, async processing, and bursty irregular traffic.
Examples: AWS Lambda, Azure Functions, Google Cloud Functions

The Shared Responsibility Model

A key concept for security interviews: as you move from IaaS → PaaS → SaaS, the cloud provider takes on more responsibility. With IaaS you own OS patching; with SaaS the vendor handles everything. You're always responsible for your data and access control.

🎯 Interview Answer Frame: "I start by asking — does our team have the operational maturity to manage this layer? If not, abstract it away. Every layer you manage yourself is engineering time not spent on business value."

⚙️ Phase 2 · Cloud Foundations
Compute: VMs, Containers & Serverless
The three compute paradigms — and how to pick the right one.

The Three Compute Paradigms

🖥️ Virtual Machines (EC2, Azure VMs)
Full OS, persistent, long-running. Boot time: minutes. Best for: stateful apps, legacy lifts, workloads needing GPU/custom OS. Downside: you pay whether idle or busy, and you manage patching.
🐳 Containers (ECS, EKS, GKE)
Lightweight, portable, share the host OS kernel. Boot time: seconds. Best for: microservices, consistent dev-to-prod environments, scaling individual services independently. Kubernetes is the de-facto orchestrator — know it exists even if you don't operate it yourself.
⚡ Serverless (Lambda, Cloud Functions)
Event-triggered, stateless, ephemeral. Boot time: milliseconds (with warm containers) to seconds (cold start). Best for: async jobs, event processing, APIs with irregular traffic. Downside: cold starts, 15-min max execution, harder to debug locally.

Decision Framework

🧭 How to Choose
Ask: Is workload event-driven and short-lived? → Serverless. Is it a microservice that needs to scale independently? → Container. Does it need a specific OS, GPU, or run continuously? → VM.
🎯 Senior SA Move: Mention that most modern architectures mix all three. Serverless for async triggers, containers for core APIs, VMs for databases or GPU workloads. Avoid the trap of picking one and applying it everywhere.

🗄️ Phase 2 · Cloud Foundations
Storage: Object, Block & File
Three storage types, three completely different use cases.

The Three Storage Types

🪣 Object Storage (S3, GCS, Blob)
Stores files as objects with metadata. Flat namespace. Accessed via HTTP/API. Infinitely scalable, cheap, durable (99.999999999% — "11 nines"). Use for: images, videos, backups, static websites, data lake raw files, ML training datasets.
💾 Block Storage (EBS, Azure Disk)
Raw storage volumes attached to a single VM. Low latency, high IOPS. Works like a hard drive. Use for: OS volumes, databases (PostgreSQL, MySQL), any app needing a mounted filesystem on one server.
📁 File Storage (EFS, Azure Files, FSx)
Shared filesystem mounted by multiple instances simultaneously. NFS-style. Use for: shared content repos, CMS media, HPC shared scratch space, lift-and-shift of apps expecting a shared drive.

Quick Decision Rule

Multiple instances need to read/write simultaneously? → File storage. One instance needs a fast disk? → Block. Storing files/objects accessed via API or URL? → Object.

🎯 Interview Pattern: "For our data lake we used S3 as the raw layer — cheap, durable, queryable via Athena. For our application databases we used EBS gp3 volumes — low latency, predictable IOPS. For our shared rendering cluster we needed EFS so all nodes could access the same asset files."

🔑 Phase 2 · Cloud Foundations
Identity & Access Management (IAM)
The foundation of cloud security. Get this wrong and nothing else matters.

The Golden Rule: Least Privilege

Every identity (human or machine) should have the minimum permissions needed to do its job — nothing more. This is the single most important IAM principle. Violating it is the root cause of the majority of cloud breaches.

🏢 Analogy
A hotel cleaner has a key card that opens guest rooms — but not the safe, the server room, or the CEO's office. IAM is your key card system. Every service, user, and role gets exactly the doors they need.

Core IAM Concepts

Users vs Roles vs Groups
Users = individual humans with long-term credentials (avoid for services). Roles = temporary credentials assumed by services, lambdas, or cross-account access (prefer this). Groups = attach policies to multiple users at once.
Policies
JSON documents that define Allow/Deny for specific Actions on specific Resources. Explicit Deny always wins over Allow. Use managed policies for common patterns, inline policies for specific exceptions.
Service Accounts & Federation
Never hardcode AWS keys in code. Use IAM Roles for EC2/Lambda (instance profiles). For human users, federate with your Identity Provider (Okta, Azure AD) via SAML or OIDC — single sign-on, central revocation.

IAM Best Practices (Interview Checklist)

Enable MFA on rootNever use root account dailyRotate access keysUse roles not users for servicesAudit with CloudTrailUse permission boundariesSCPs for org-wide guardrails
🎯 Senior SA Answer: "I'd enforce least privilege via IAM roles with permission boundaries, federate human access through our IdP, enable CloudTrail for all API calls, and use AWS Config rules to continuously audit for overly permissive policies — flagging any * wildcard actions automatically."

What is a Solutions Architect?
The mindset, responsibilities, and superpowers of your role.

The Role in One Sentence

A Solutions Architect is the translator between what the business needs and what technology can deliver — making complex technical decisions that have real commercial consequences.

🏗️ Real World Analogy
Think of yourself as an architect designing a skyscraper. You don't lay every brick yourself — but you make the critical decisions: structural load, materials, safety systems, and aesthetics. The builders (engineers) execute your plan. If your plan is wrong, the building falls. If it's brilliant, it stands for 50 years.

Your Core Responsibilities

🎯
Design
Create technical blueprints that solve business problems
🗣️
Communicate
Translate tech to execs and business to engineers
⚖️
Trade-offs
Balance cost, speed, reliability and security
🔮
Anticipate
Design for tomorrow's scale, not just today's
🛡️
Risk
Identify and mitigate technical and business risks
🤝
Align
Ensure technical roadmap matches business strategy

Junior vs Senior Mindset

Junior SA thinks: "What's the best technology for this?"
Senior SA thinks: "What's the best outcome for the business — and which technology gets us there with acceptable risk and cost?"

The shift is from technology-first to outcome-first thinking. This is the single most important mindset upgrade in your journey.

🎯 Interview Gold: When asked a design question, always start by asking about business requirements, scale expectations, budget constraints, and acceptable downtime. Interviewers reward this instinct.

The SA's Non-Functional Requirements (NFRs)

These are the invisible requirements that define your architecture quality. Memorise these — they appear in every design discussion:

ScalabilityAvailabilityReliabilityPerformanceSecurityMaintainabilityObservabilityCost-efficiencyCompliancePortability

Senior architects don't wait to be asked about NFRs — they proactively raise them. Every design decision should be traceable back to one of these.

⚖️ Lesson 2 · Architecture Fundamentals
The Well-Architected Framework
The 6 pillars that define great architecture. A senior interview staple.

Why This Matters

AWS's Well-Architected Framework is the closest thing to a universal standard for evaluating cloud architectures. Even if you work on Azure or GCP, the principles translate. Senior interviewers use these pillars to structure technical conversations.

🎯 Pro Tip: In any system design interview, you can score major points by saying "Let me walk through this from a Well-Architected perspective..." — it signals structured, mature thinking.

The 6 Pillars

1. 🔧 Operational Excellence
Running and monitoring systems to deliver business value, then continually improving processes. Think: IaC, CI/CD, runbooks, post-mortems, and small frequent changes over big risky releases.
2. 🛡️ Security
Protecting data, systems, and assets. Key principles: least privilege, defence in depth, encryption everywhere, automated security checks, and treating security as code — not an afterthought.
3. 🔄 Reliability
The ability to recover from failures and meet business demand. Covers: circuit breakers, retries with exponential backoff, multi-AZ deployment, automated failover, and chaos engineering.
4. ⚡ Performance Efficiency
Using resources efficiently as demand changes. Know when to scale horizontally vs vertically, when caching helps, and how to benchmark. "Right-size" is a senior architect phrase.
5. 💰 Cost Optimization
Avoid unnecessary spend. Understand CapEx vs OpEx, Spot vs Reserved vs On-Demand, and always tag your resources for cost visibility. FinOps is a growing discipline.
6. 🌱 Sustainability
The newest pillar. Minimize environmental impact — choose regions with renewable energy, right-size to reduce waste, optimize for energy efficiency. ESG-conscious companies ask about this.

How to Apply This in Practice

A Well-Architected Review (WAR) is a structured conversation you run with a team — typically using AWS's questionnaire tool — that produces a report of high-risk issues (HRIs) and medium-risk issues (MRIs). As a senior SA you'll run these for existing systems.

🧭 Mental Model
Think of the 6 pillars as 6 lenses. Every architecture decision you make, look through each lens: "Does this choice make me more or less operationally excellent? More or less secure?" You won't always optimise all 6 simultaneously — that's where trade-offs come in.

🎭 Lesson 3 · Architecture Fundamentals
The Art of Trade-offs & CAP Theorem
The concept that separates junior from senior thinking.

There Is No Perfect Architecture

This is the most important sentence in architecture. Every design decision is a trade-off. The junior SA looks for the "right answer." The senior SA asks "right for whom, when, and at what cost?"

CAP Theorem — Explained Simply

In any distributed data system, you can only guarantee TWO of these three properties simultaneously:

🔒
Consistency
Every read gets the most recent write (or an error)
⚡
Availability
Every request gets a response (maybe not the latest data)
🌐
Partition Tolerance
System continues despite network partitions
The key insight: Network partitions are unavoidable in distributed systems. So you're really choosing between CP (consistent but may be unavailable) and AP (available but may return stale data). This is a business decision, not just a technical one.

Real-World Examples

Choose CP (banking, payments, inventory): A bank account balance must be consistent. Better to return an error than to show a wrong balance. DynamoDB in strong consistency mode, PostgreSQL with synchronous replication.
Choose AP (social feeds, product catalogs, DNS): If Twitter is temporarily inconsistent — showing a like count that's 2 seconds stale — nobody cares. But if Twitter goes down, millions are affected. Cassandra, DynamoDB in eventual consistency mode, Couchbase.

The PACELC Extension

PACELC goes further: even when there's NO partition, you still face a trade-off between latency and consistency. Lower latency = more stale data possible. This is why global databases are genuinely hard.

🎓 Interview Tip
Mention PACELC after CAP in interviews — it signals you've gone deeper than most. Say: "CAP is the partition scenario, but PACELC reminds us that even under normal operation, latency and consistency compete."

📐 Lesson 4 · Architecture Fundamentals
Architecture Diagrams & Communication
The C4 Model and how to communicate designs like a senior.

Why Diagrams Are a Superpower

A great architecture diagram can align a 20-person team in 10 minutes. A bad one creates confusion for months. As an SA, your ability to communicate visually is as important as your technical knowledge.

The Golden Rule: Different audiences need different levels of abstraction. An executive needs a context diagram. A developer needs a component diagram. Never show the same diagram to both.

The C4 Model — Your Go-To Framework

C4 stands for Context, Container, Component, and Code. It gives you 4 zoom levels for any system:

🌍
L1: Context
Show the system + its users + external systems. For executives and stakeholders.
📦
L2: Container
Apps, databases, microservices, file stores. For senior engineers and architects.
⚙️
L3: Component
Internal components of one container. For the team building that service.
💻
L4: Code
Classes, functions. Rarely needed — usually auto-generated. For devs only.

Tools You Should Know

draw.io (free) Lucidchart Miro Structurizr (C4 native) PlantUML (code-based) Excalidraw (whiteboard)
🎯 Interview Whiteboard Tip: In system design interviews, always start by drawing the context diagram before going into detail. Say "Let me establish the system boundary first." This buys you thinking time AND shows structured thinking.

The 3 Audiences & What They Need

Executives / C-Suite: Context diagram only. No technical jargon. Focus on: what the system does, who uses it, and what business value it delivers. Use plain English labels.
Product / Business Analysts: Container level. Explain flows in terms of user journeys. "When a customer checks out, the Order Service calls the Payment Service, which calls Stripe."
Engineers: Component level + explicit protocols (REST, gRPC, async queue). Include data formats, ports, auth mechanisms. This is where technical precision matters.
